# Releasing Vesta

Vesta ships as a **signed Android APK attached to a GitHub Release**. Pushing a
`v*` tag runs `.github/workflows/release.yml`, which builds `assembleRelease`
signed with the upload keystore and uploads the APK. Play Store / F-Droid
distribution is out of scope (see GAMEPLAN Fase 5).

`android/` is generated by `expo prebuild` and gitignored, so the release
signing config lives in the `with-android-signing` config plugin
(`apps/mobile/plugins/with-android-signing.js`). It points the release build
type at the upload keystore **only when the `VESTA_UPLOAD_*` Gradle properties
are set** (the release workflow sets them from secrets); otherwise it falls back
to the debug keystore, so local and CI debug builds are unaffected.

## One-time setup (repo owner)

You need a release keystore and four repository secrets. **Do this once. Keep
the keystore and its passwords safe** — losing them means you can never publish
an update signed with the same identity.

### 1. Generate the upload keystore

```bash
keytool -genkeypair -v \
  -keystore vesta-upload.jks \
  -alias vesta-upload \
  -keyalg RSA -keysize 2048 -validity 10000 \
  -storepass '<STORE_PASSWORD>' -keypass '<KEY_PASSWORD>' \
  -dname "CN=Vesta, O=Cosmico, C=IT"
```

Store `vesta-upload.jks` somewhere safe and private. **Do not commit it** — the
repo's `.gitignore` already excludes `*.jks`/`*.keystore`.

### 2. Set the four repository secrets

```bash
# From the repo root, with the gh CLI authenticated:
base64 -i vesta-upload.jks | gh secret set ANDROID_KEYSTORE_BASE64
gh secret set ANDROID_KEYSTORE_PASSWORD   # paste <STORE_PASSWORD>
gh secret set ANDROID_KEY_ALIAS           # enter: vesta-upload
gh secret set ANDROID_KEY_PASSWORD        # paste <KEY_PASSWORD>
```

Verify with `gh secret list` — all four must be present. The release workflow
fails fast with a clear message if any is missing.

## Cutting a release

1. Make sure `main` is green and `apps/mobile/app.json` has the intended
   `version` and `android.versionCode` (bump `versionCode` for every release —
   Android rejects an install that doesn't increase it).
2. Move the `CHANGELOG.md` `[Unreleased]` section under a new
   `## [x.y.z] — <date>` heading.
3. Tag and push:

   ```bash
   git tag v0.2.0
   git push origin v0.2.0
   ```

4. The `Release` workflow builds the signed APK and creates the GitHub Release
   with auto-generated notes and `vesta-v0.2.0.apk` attached. Edit the release
   notes if you want the CHANGELOG text instead.

## How the workflow signs

- `expo prebuild` regenerates `android/` and runs `with-android-signing`, which
  adds a `release` signing config and switches the release build type to it when
  `VESTA_UPLOAD_STORE_FILE` is present.
- The workflow decodes `ANDROID_KEYSTORE_BASE64` into
  `android/app/release.keystore` and passes the passwords/alias as
  `ORG_GRADLE_PROJECT_VESTA_UPLOAD_*` env vars (Gradle reads these as project
  properties — no secrets on the command line or in files), then runs
  `assembleRelease`. The decoded keystore is removed from the runner afterward.
